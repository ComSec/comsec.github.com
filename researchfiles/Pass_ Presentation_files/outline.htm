Problem Definition
  • Current authentication technologies:
    • Rely on public key technique
    • Require lots of computational resources
      • Processing Power & Storage


  • Lightweight mobile devices (PDA, cell phone, etc):
    • Lack the computational resources needed to use public key technique directly

5
Proposed Solution
  • Proxy-Assisted Security Service (PASS)
    • Implementation of the protocol
    • Move the computational intensive parts of mutual authentication to Security Proxies

Design Decisions (1)
  • .NET Framework vs Java
  • Decision Made:  .NET Framework
    • Pros
      • Pocket PC natively supports .NET Framework
      • .NET provides many security features
      • Our entire team has experience with Windows Platform development
    • Cons
      • Less portable
12
Design Decisions (2)
  • Microsoft SQL Server vs Microsoft Access as the database engine
  • Decision made: Microsoft SQL Server
    • Pros
      • Supports clustering and advanced database features such as transactions and stored procedures
    • Cons
      • More programming skills required
      • More maintenance
      • Higher footprint
13
Design Decisions (3)
  • I/O Completion Ports model vs One thread per connection
  • Decision Made:  I/O Completion Ports model
    • Pros
      • I/O Completion Ports design minimizes the problem of threads blocking at synchronous network operations, which improves performance scalability and resource usage
    • Cons
      • More complex design
      • Harder to debug

Technical Difficulties
15
Difficulties
  • Portability Issues
    • Keeping multiple versions of source code for different platforms is problematic
    • Goal: Code for 100% portability
  • Performance Issues
    • Equally important for both the Security Proxy and for the Client-Side API
16
Portability Issues (1)
  • Problem 1: Maximizing code reuse between the Pocket PC and the desktop version
    • Solution: Try to write as much .NET code as possible


  • Problem 2: The .NET API for Pocket PC does not support the threading model originally proposed for performing network I/O
    • Solution: Re-design to use a delayed polling mechanism. No performance penalty expected because this is only for client-side code
17
Portability Issues (2)
  • Problem 3: The .NET API for Pocket PC does not support cryptography functions at all (the desktop version does)
    • Solution: Fortunately, both the Win32 API and the Pocket PC 2003 API support the same set of cryptography functions. We are able to use these functions using C++ code that is still 100% portable.

  • Problem 4: The Diffie-Hellman variant that we need is not supported by the Win32 API
    • Solution: 3rd-party libraries that provide this capability exist, but are not portable. Finally, we wrote this ourselves. (Difficult, and caused performance issues)
18
Portability Issues (3)
  • In the end, over 95% of our code are fully portable, without the use of conditional compiling. The remaining are UI code
  • We have picked the right API, namely the .NET API
  • Could have saved some time if we researched more on the differences between the .NET API for Pocket PC and the desktop, before writing the code!
19
Performance Issues (1)
  • Problem: The Diffie-Hellman system implemented by us is almost 4x slower than a 3rd-party library. Profiling shows that this is the bottleneck of the entire system
    • Solution: Research shows that our original approach, which is based on multi-precision integer division, is very slow. After switching to use another algorithm based on Montgomery Multiplication, and after some code tuning for optimizing memory access, performance is on par with 3rd party library
20
Performance Issues (2)
  • In the end, performance requirement is met


  • More speedup is possible by an assembly language implementation of the Diffie-Hellman algorithm, but will cause portability issue

Test Results (1)
  • Settings
    • SP=Security Proxy, MS=Mobile Station, DS=Directory Service
    • The PC used is a P4 2.4Ghz machine, with 512MB RAM
    • This PC hosts both the SP and the DS
    • There is only one SP in the system
    • This PC also hosts the Client-Side component whenever Mobile Station 1 is to be run on a PC
    • The Pocket PC is a Dell AXIM X5, with an X-Scale 300 MHz processor
    • The Pocket PC communicates with the PC via a 802.11b wireless network (no encryption)
    • For each case, we perform 4 trials. The final result is taken as the arithmetic average of the last 3 trials
    • 512-bit Diffie-Hellman key size

Test Result Summary
